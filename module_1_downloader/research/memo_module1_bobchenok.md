# Модуль 1 — Сбор и скачивание статей

**Ответственный:** Арсений Бобченок (Бобан)
**Оценка:** ~1 неделя, 1 человек
**Статус:** research → dev

---

## Цель модуля

На вход — список/словарь ключевых слов по химической тематике.
На выход — папка скачанных PDF-статей + манифест с метаданными.

Это самый первый этап пайплайна: он «кормит» Модуль 2 (парсер PDF→JSON).

**Цепочка:**
`keywords → CrossRef (поиск, получение DOI) → Sci-Hub (скачивание по DOI) → PDF + manifest.json → вход Модуля 2`

---

## Границы (scope)

**Входит в модуль:**
- поиск статей по ключевым словам через CrossRef;
- получение DOI + метаданных;
- скачивание PDF по DOI через Sci-Hub;
- сохранение файлов + манифест в согласованном формате.

**НЕ входит (это Модуль 2):**
- парсинг PDF, извлечение текста/таблиц/картинок/формул;
- перевод в JSON.

Мой выход должен без переходников заходить во вход парсера.

---

## Что переиспользую из тестового (`pdf-to-json-parser`)

- **`metadata.py` → `query_crossref(doi)`** — готовый паттерн запроса к CrossRef (`httpx`, `User-Agent`, polite pool через `mailto`, таймаут, обработка ошибок). Это моя же наработка, беру за основу.
- **DOI-утилиты**: `DOI_RE`, `find_doi()`, `_is_placeholder_doi()`, нормализация — для чистки/валидации DOI из поиска.
- **Инфра/стиль проекта**: `uv` + `pyproject`, CLI на Typer, логирование loguru, схемы Pydantic, Docker/compose. Зеркалю, чтобы всё было единообразно с командой.

> Важно: существующий `query_crossref` делает лукап по **известному** DOI (`GET /works/{doi}`). Мне нужен обратный поиск — по ключевым словам. Это дописываю сам.

## Тайтлы (задачи)

### T1. Research: CrossRef API
- разобрать поиск по ключевым словам: `GET /works?query.bibliographic=...`;
- фильтры: `type:journal-article`, `from-pub-date`/`until-pub-date` и т.п.;
- глубокая пагинация через `cursor=*` (не `offset`);
- лимиты и polite pool (`mailto`).
- **Готово, когда:** есть заметка/ноутбук с рабочими примерами запросов и понятной схемой ответа.

### T2. Research: Sci-Hub
- как скачивать по DOI, работа с зеркалами;
- ограничения (статьи примерно до 2024 г. — по словам Никиты);
- легальность: только научные/некоммерческие цели, открытый код желателен.
- **Готово, когда:** подтверждено, что по DOI можно стабильно получить PDF, описаны кейсы «нет статьи».

### T3. Модуль поиска — `crossref_search.py`
- вход: список/словарь keywords → выход: список DOI + базовые метаданные;
- применение фильтров, дедуп по DOI, обработка пустых результатов.
- **Готово, когда:** по тестовой теме отдаёт валидный список DOI.

### T4. Модуль скачивания — `scihub_downloader.py`
- DOI → резолв зеркала → достать ссылку на PDF (парсинг HTML/`embed`) → скачать на диск;
- ретраи, троттлинг, аккуратная обработка «статья не найдена».
- **Готово, когда:** по списку DOI кладёт PDF-ы в папку, пропущенные логирует.

### T5. Оркестрация + манифест
- связать поиск и качалку в один поток;
- сохранять PDF + `manifest.json` (DOI, метаданные, ключевое слово, статус скачивания);
- манифест под Pydantic-схему, совместимую со `schemas.py` парсера.
- **Готово, когда:** одна команда `keywords → папка PDF + manifest`.

### T6. CLI + конфиг
- Typer-команда в стиле `cli.py` из репы;
- конфиг: `mailto`, зеркала Sci-Hub, rate-limit, фильтры по датам.
- **Готово, когда:** запускается из консоли с параметрами, логи через loguru.

### T7. Тесты + валидация
- прогон на реальных темах (напр. «oxidation induction time», «antioxidants»);
- проверка, что выход заходит в парсер Модуля 2.
- **Готово, когда:** end-to-end на 1–2 темах отрабатывает.
